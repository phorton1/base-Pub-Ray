**What it is.** mod001 adds a remote read-memory / write-memory / call-function
("peek / poke / call") primitive to the running E80 over Ethernet/UDP, by re-purposing the
firmware's unadvertised **Diagnostics** service (UDP 6667). It reclaims the silent
CHECKSUM-VERIFY command's two helper routines -- which have no wire response and no side
effect -- and routes that command into the service's existing echo reply path, so no new
transport code is added. The edit is **length-preserving** (every overwritten byte is
positively known to be dead; nothing is added or removed).

**What it enables.** This is the live-device channel the project reads and writes real
hardware with. Its full design -- the three operations, the wiring, the wire protocol, and
what it has settled -- is documented as the project's instrument in
[diagnostics](diagnostics.md); it is the substrate under the config
backup/restore library ([cleanroom/e80Config_API](../cleanroom/e80Config_API.md)). It is a
debug primitive, not a feature in itself.

## Code-write locations

Length-preserving (no byte added or removed). Two regions of known-dead code are given up:

| record | location | size | what |
|---|---|---|---|
| handler | `0x0053a808` | 260 B | the dead `verifyChecksumCmd` + `computeXorChecksum` helpers (`0x0053a808-0x0053a90b`), overwritten with the hand-assembled peek / poke / call / call5 handler -- 65 dense ARM instructions, no NOP pad, including a shared byte-to-hex encoder and a set-length tail |
| splice | `0x0053af78` | 32 B | the silent `0xdddd0005` branch, rewritten to call the handler then fall into the existing ECHO reply path -- 3 instructions + a 5-instruction NOP pad |
| version | `0x0017a924` + `0x0017a93c` | 2x4 B | the SysInfo build-stamp self-check + fallback, bumped `569 -> 571` so a standalone v5.71 package reports its own version (see Version) |

The two helpers' **only** cross-references are each other and the silent branch (confirmed by
cross-reference), so the 260-byte region is safe to reclaim. The dispatch, the N2K-insertion
command (`0xdddd0006`), and the ECHO command (`0xdddd0000`) are untouched and still work.

## How it wires in (no new transport code)

The `0xdddd0005` (checksum-verify) command was **silent** -- it validated a packet and stored a
bool on the stack, sending nothing -- so its branch and its two helpers were pure dead weight.
mod001 reuses the service's own machinery rather than adding a transport:

- The **splice** rewrites that silent branch to `add r0, sp, #0x58` (a pointer to the listener's
  request/reply buffer), `bl` the handler, then `b` into the existing **ECHO reply path**.
- The **ECHO path** already builds and sends a reply: it copies a payload from `recvbuf+8`
  (length at `recvbuf[6..7]`) to the requester at the reply port in `recvbuf[4..5]`. So the
  handler only has to **fill `recvbuf+8`** with its result and set the length; control then falls
  into the reply path, which sends it.
- `r4` (held `=0` by the listener) and `r5` (`=this`, the server object) are **callee-saved** by
  the handler, so the ECHO path resumes intact.

## Wire protocol (device side)

This is the device-side contract; the operational view -- how the project drives it -- is the
project instrument, [diagnostics](diagnostics.md). A request to UDP **6667** is a
native `0xdddd` datagram, all fields word-aligned:

```
[0..3]   0xdddd0005          dispatch (rides the kept listener path)
[4..5]   reply UDP port      where the client listens
[8]      subop               0 = peek, 1 = poke, 2 = call
[12..15] address / funcptr   memory address (peek/poke) or function pointer (call)
  peek:  [16..19] count                         -> reply = count bytes from the address, hex-encoded
  poke:  [16..19] count; [20..] bytes to write   -> reply length 0 (verify with a follow-up peek)
  call:  [16..35] arg0..arg4 (five 32-bit args)  -> reply = 4-byte return value, hex-encoded
```

The reply is built by the ECHO path: a 10-byte header (`0xdddd0100`, server id, payload length)
then the payload the handler wrote. `peek` and `call` outputs are hex-encoded by the handler's
shared byte-to-hex encoder; `call` passes its fifth argument on the stack per the ARM calling
convention.

## Version

mod001's binary is built by copying the stock image to a per-mod working binary, applying record 001, and packaging it (see
[modification](modification.md)); the standalone package is labeled **v5.71**. The running
app's reported version is a
**self-check** against a hardcoded constant in the SysInfo build-stamp populator (the mechanism
mod002 documents in detail), so the **version** records bump that constant `569 -> 571`; a v5.71
package then reports its own version instead of falling back to "569 / Unknown". When mod001
ships inside the **v5.72** mod002 package, mod002's version record chains it `571 -> 572`
(see [mods](mods.md)). The read-side layout of that stamp -- where a peek
client reads the running version, model, and build fields out of the
SysInfo singleton -- is in [architecture/runtime](../architecture/runtime.md).

## History -- the name was reused

The *first* mod001 was a single same-width ARM instruction edit that flipped a firmware enable
gate. It was built through the [modification](modification.md) pipeline, installed, and
**booted on hardware** (2026-05-31) with the user config retained -- the first end-to-end proof
of the build/flash scheme. But what it unlocked turned out to be a dead end (see
[abstracts/TFTP](../abstracts/TFTP.md)), so that mod was **reverted** and the `mod001` slot was
re-scoped to the diagnostics primitive described here. The build mechanics are proven
regardless of what any given mod targets.

## Recovery

A bad address can fault the listener thread -- it is a debug primitive used on owned
hardware -- and the backstop is the stock-package reflash, the recovery floor in
[modification](modification.md).
