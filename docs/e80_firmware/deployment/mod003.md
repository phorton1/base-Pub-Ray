**What it is.** mod003 makes the E80's on-device track recorder stamp each committed
track point with a real **wall-clock time** -- which a stock point does not carry -- and,
in the same point, the **true depth in feet**. A stock track point holds position +
depth + water temperature but no time. The default for this mod is to write a "timed"
track unless a particular Database value is set to a nonzero value, in which case it writes a stock track.

A stock track comes back with no record of *when* it was logged. mod003
fills that gap. It is **length-preserving** and **format-preserving**: it rides the
existing 14-byte track-point record and overloads two of its fields rather than changing
any structure, so the storage, the saved-track database, and the network read/write path
are all untouched. A **DB toggle** gates the behaviour: a client can set it (nonzero) to make a modded
unit record stock tracks without any reflash -- and reflashing stock Raymarine firmware,
of course, removes the mod altogether.

**What it enables.** Date-stamped tracks for the standalone-recording case that motivated
it: the E80 logs a passage by itself, and a later reader -- navMate, or any
[TRACK](../abstracts/TRACK.md) client -- recovers the time and depth of every point
directly from the point stream. No track-type flag and no side channel: a timed track is
self-identifying (below).

## Code-write locations

Length-preserving throughout (the mod001 / mod002 model): no byte added or removed, every
overwritten byte positively known to be free.

| record | location | size | what |
|---|---|---|---|
| **[A] emit-site retargets** | `0x002f86c8`, `0x002f8eb4`, `0x002e98e4` | 3x4 B | the `bl TrackRecorder_addPoint` (`0x002e8c34`) in each of the three track calculators -- Auto (`CAutoTrackCalculator`), Distance, Time -- retargeted to `bl mod003_stub`. Only *real* points reach addPoint (segment breaks bypass it), so only real points route through the stub |
| **[B] mod003_stub** | `0x00293340` | 304 B | the toggle-gated datetime/depth injector, in a reclaimed **InterNiche dev-CLI** reservoir (run 06, `0x00293340-0x0029346f`; the run's spare tail leaves a 68 B margin). A transparent wrapper around addPoint |
| **[D] version stamp** | `0x0017a924` + `0x0017a93c` | 2x4 B | the SysInfo build-stamp self-check + fallback, `572 -> 573` (chained from mod002's `571 -> 572`; see Version) |

## How it wires in

mod003_stub is a **transparent wrapper** around `TrackRecorder_addPoint`: the three emit
sites now `bl` the stub instead, and the stub either tail-calls addPoint with the point
**untouched** or stamps the point first and then tail-calls. The decision is made **per
point**, at the head of the stub, by reading the toggle (below):

- **toggle set (nonzero):** branch straight to the epilogue and tail-call addPoint with
  the point unchanged -- a stock, non-timed track.
- **toggle 0 or absent (the default):** read the day and time data items, pack them into
  the point, and tail-call addPoint -- a timed track.

The injector reads the day/time items through the firmware's own data-provider singleton
(its `getItem` vmethod) rather than carrying its own; the indirect call uses the
`mov lr,pc; bx ip` form this ARMv4T core requires (it has no `blx`). Routing all three
calculators means every recording mode -- Auto, Time, Distance -- produces timed tracks
identically.

## The timed-track contract

This is what a **reader** sees. A track point is 14 bytes:

```
north  (4)   prescaled Mercator northing
east   (4)   prescaled Mercator easting
temp   (2)   water temperature (Kelvin)   -- stock
depth  (4)   depth, centimetres           -- stock
```

In a **timed** track, the recorder overloads the last two fields:

```
temp   (2)   real depth in 0.1 ft   = (depth_cm * 0x5400) >> 16   (cm -> tenths of a foot)
depth  (4)   unix timestamp (sec)   = (date & 0xffff) * 86400 + time / 10000
```

where `date` is the device's day-count data item and `time` is its time-of-day item in
0.1 ms units (hence `/10000` to seconds). A present-day value lands near **1.78e9**.

**Detection is by value, not by a flag.** Read the `depth` field as a raw little-endian
u32 and interpret it as a unix timestamp: if it is **>= 1980-01-01** -- that is,
`>= 315532800` (`0x12CEA600`) -- the point is timed, and that field holds a date, not a
depth. **Otherwise** -- the u32 is below that threshold -- the field is an ordinary depth
in centimetres and the point belongs to an original, non-timed track. The two readings
never collide: no real depth in centimetres approaches 315
million (~3155 km), and every real recording time is long after 1980, so the threshold
separates them with vast margin (a present-day timed value lands near 1.78e9). No
track-type bit is set, and the on-wire record is byte-identical to a stock one. **Segment breaks are unchanged:** a recording stop still
writes the all-ones sentinel point (`north = east = 0xFFFFFFFF`, `temp = depth =
0xFFFF`), which the stub never sees (segment breaks do not pass through addPoint), so the
existing segment-break test still holds.

A reader therefore: treats an all-ones point as a segment break; treats a point whose
raw `depth` u32 is `>= 315532800` (a unix time at/after 1980-01-01) as a timed point
(time in `depth`, depth-in-0.1ft in `temp`); and otherwise treats it as an original
normal track point (depth-cm in `depth`, temperature in `temp`).

## The toggle

The behaviour is gated by a single application-namespace database item:

| fid | ENC | type | bytes |
|---|---|---|---|
| `0x05ff0001` | `0x0054` | BOOL8 | 1 |

It is a **novel item** minted over the stock DATABASE protocol -- not a factory setting,
so it carries no Setup-menu label -- and it lives in the persistent application namespace
(it survives reboot). It is **opt-out / default-on**:

- **absent or 0** -> timed tracks. The stub's read returns 0 for both "explicitly 0" and
  "never minted", so a unit that has never had the item written records timed tracks by
  default.
- **nonzero** -> stock, non-timed tracks.

To disable the feature, **mint the item and set it nonzero** with an ordinary
point-to-point `SetItem` write over the stock protocol (see
[DATABASE](../abstracts/DATABASE.md) and [DB_FIDS](../abstracts/DB_FIDS.md)); to
re-enable, clear it to 0. navMate exposes this in its UI. The mod itself only **reads**
the item -- it never writes it.

## Version

**v5.73**; the shipped package carries **mod001 + mod002 + mod003** (see [mods](mods.md)).
The reported app version is a self-check against a hardcoded constant in the SysInfo
build-stamp populator (the mechanism [mod002](mod002.md) documents); mod003 builds on the
mod002 binary, so its `[D]` records chain that constant `572 -> 573` and the package
reports v5.73. The stamp is display + installer-ordering only.

## Recovery

The stub runs on the recorder's own path; a fault there affects track recording, not the
rest of the system, and -- as with mod001 / mod002 -- the backstop is the stock-package
reflash, the recovery floor in [modification](modification.md).
