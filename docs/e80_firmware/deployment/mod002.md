**What it is.** mod002 adds a fast, tear-free full-screen **capture** to the running
E80. A host sends one GRAB command to the Diagnostics **TCP** port (6668); the device
takes a raw **atomic snapshot** of the display hardware's state -- the GDC2 graphics
register block, its colour tables, and each enabled layer's raw buffer -- into a heap
scratch buffer and streams it back over the same connection. All compositing and PNG
encoding happen on the **host**; nothing is converted on the device, which is what keeps
the mod small. It also stamps the running app's reported version to 5.72.

**What it enables.** The live-screen capture library
([cleanroom/e80ScreenGrab_API](../cleanroom/e80ScreenGrab_API.md)) -- read-only and
resolution-agnostic (640x480 on an E80, 800x600 on an E120). That doc is the
host-implementation view of the wire contract given below.

## Code-write locations

Length-preserving throughout (the mod001 model): no byte added or removed, all downstream
offsets unchanged. Each region is **known dead code** we give up:

| record | location | size | what |
|---|---|---|---|
| **[A] gate stub** | `0x0029d51c` | 8 B | entry of the dead TFTP handler -- made to return immediately |
| **[B] grab_routine** | `0x0029d524` | 420 B | the snapshot routine, in the reclaimed handler body (`0x0029d524-0x0029d6c7`; hard ceiling `0x0029d7f4`) |
| **[C] dispatch splice** | `0x0059fec0` | 512 B | the conn-thread echo window, repurposed to recognise GRAB |
| **[D] version stamp** | `0x0017a924` + `0x0017a93c` | 2x4 B | the SysInfo build-stamp self-check + fallback, `571 -> 572` (chained from mod001's `569 -> 571`; see Version) |

## Why the TFTP reclaim ([A]+[B]) is safe

`TFTP_requestHandler` (`0x0029d51c .. 0x0029d8f7`, 988 B) is a **dead end**: its file
layer serves no files (see [abstracts/TFTP](../abstracts/TFTP.md)), it is gated off behind
an enable flag, and the reverted first mod001 already proved that flipping that gate
reaches nothing useful. So the whole body is reclaimed:

- **[A]** the entry stub returns immediately (mimicking the stock "transfers disabled"
  path), so the TFTP path -- even if the server is listening -- can never fall into
  grab_routine's bytes. This nails the gate shut.
- **[B]** grab_routine (420 B) occupies `0x0029d524-0x0029d6c7`, entered **only** via a
  `BL` from the dispatch in [C]. There is a **hard size ceiling at `0x0029d7f4`** (720 B
  max): the old handler's tail holds a gate-pointer literal at `0x0029d7f4` that the
  surviving `TFTP_serverStart`/`TFTP_serverStop` still read -- grab_routine must never
  grow past it. At 420 B the margin is 300 B. The body start has
  no xrefs, and the next function -- the boot-time socket helper at `0x0029d8f8` -- borrows
  nothing from this region; both confirmed by cross-reference.

## Why the conn-thread echo window ([C]) is safe

`CLNetDiagnosticsConnection_thread` (`0x0059fe6c`) is **echo-only**: every non-echo command
is read and silently dropped. Its reply path (`0x0059fec0..0x005a00bc`, 512 B / 128
instructions) is the echo build+send, which we discard -- there is no living echo client
(mod001's UDP 6667 already provides liveness). The spans **around** it are kept intact, so
control flow is preserved:

| span | role | disposition |
|---|---|---|
| `0x0059fe6c-0x0059fe84` | prologue (sets up the conn object) | keep |
| `0x0059fe88-0x0059feb0` | recv one full frame (vt+0x24, cap `0xb54`) | keep |
| `0x0059feb4-0x0059febc` | recv-result check | keep |
| `0x0059fec0-0x005a00bc` | echo build+send | **replace -> dispatch [C]** |
| `0x005a00c0-0x005a00d0` | loop condition (branch target) | keep |
| `0x005a00d4-0x005a00ec` | connection reap + epilogue | keep |

The dispatch uses ~12 instructions of the 512 B window; the remainder is NOP-padded.

## Wire format

The host-implementation view is in
[cleanroom/e80ScreenGrab_API](../cleanroom/e80ScreenGrab_API.md); this is the device-side
contract.

**Request** -- a native `0xdddd` message (rides the kept recv unchanged); the host prepends
the socket layer's 2-byte little-endian length prefix, which the recv wrapper strips before
delivering the frame:

```
[u16 LE prefix = 12]      socket-layer framing (stripped by recv)
[0..3]  cmd  = 0xdddd0007  GRAB (retires the stock echo word 0xdddd0000)
[4..5]  slot = 0           native UDP reply-port slot; unused on TCP
[6..7]  len  = 4           native u16 payload length
[8..11] seq  (u32 LE)      the host library's increasing sequence
```

**Reply** -- our own bulk stream, *not* a native message. The native send wrapper (vt+0x14)
prepends a u16 prefix that cannot express ~1.4 MB, so grab_routine bypasses it and calls the
raw stream send directly, with its own header. All fields u32 LE:

```
HEADER (16 B):
  [0..3]   cmd      = 0xdddd0007   echo of GRAB (host confirms the reply)
  [4..7]   seq                     echoed from the request (host correlates)
  [8..11]  len                     BODY length in bytes (32-bit -> carries MBs)
  [12..15] checksum                32-bit additive word-sum over the BODY
BODY (len bytes):
  [0..3]   format = 1              snapshot layout version
  [4..]    regblock dump           0x1800 B from 0xD7FD0000 (regs + the 4 CLUT banks)
  [..]     each ENABLED layer's raw buffer, in ascending layer-index order
```

The checksum is a simple additive 32-bit word-sum (there is no crc32 in the firmware --
confirmed by search -- and the only threat is a device-side assembly bug; TCP guards the
wire). The BODY is 4-aligned, so the sum needs no byte tail. The host walks the regblock's
enable mask to know how many layer buffers follow and each one's size.

## Reused functions and resolved offsets

The grab routine calls existing firmware routines rather than carrying its own:

| routine | address | signature / note |
|---|---|---|
| `rm_malloc` | `0x002a9434` | `(nbytes) -> ptr \| 0` (general heap) |
| `rm_free` | `0x001794e4` | `(ptr) -> void` (paired with rm_malloc) |
| `InterNiche_streamSend` | `0x002a5d80` | `(handle, buf, len, flags) -> total_sent \| -1`; errno at `handle+0x28` (`EWOULDBLOCK=0xb`); may return PARTIAL, so the caller loops |
| `rm_threadSleep` | `0x0016f594` | `(ms)` -- the EWOULDBLOCK transmit backoff |

The socket handle is `*(u32*)(sockobj + 4)`, where `sockobj = conn[1] = param_1[1]`; the
framed send wrapper `ConnSocket_sendFramed` (`0x00b28b1c`) is **bypassed** because of its
u16 prefix.

The GDC2 display hardware (CPU-side, fixed):

- register block at `0xD7FD0000`; a layer's base maps to CPU as `base_reg + 0xD6000000`.
- enable register at `+0x100`: bit `(16 + N)` set means HW layer N is enabled.
- per-layer control offsets `{0x20, 0x30, 0x40, 0x58, 0x70, 0x88}`; each layer's base
  offset is its control offset `+ 4`.
- control-word fields: bit31 = 16bpp; bits16-23 = pitch in tiles; low 12 bits = height-1.
- layer byte size = `((ctrl>>16 & 0xff) << 6) * ((ctrl & 0xfff) + 1)` (bytes/row x height;
  bytes/row = tiles<<6 for both bit depths).
- CLUT banks at regblock `+ {0x400, 0x800, 0x1000, 0x1400}` (inside the 0x1800 dump).

## Runtime note -- scratch sizing

The full-frame snapshot allocates ~1.4 MB on the general heap in one `rm_malloc`. This is
the one item not statically verifiable; if a contiguous 1.4 MB allocation ever proves
unreliable on-device, the documented fallback is **per-layer streaming** (allocate only the
largest single layer, ~614 KB; copy and send each layer in turn) with a trailing checksum --
host-transparent.

## Version & hardware confirmation

**v5.72**; the shipped package carries **mod001 + mod002** (see [mods](mods.md)). The reported
version is a self-check against a hardcoded constant in the SysInfo build-stamp populator; mod001
already bumps it `569 -> 571`, and because mod002 builds on the mod001 binary, its `[D]` records
chain that constant `571 -> 572` -- the one place the two byte-disjoint mods touch the same words.
Proven end-to-end on a real E80: a single GRAB returned a full multi-layer snapshot that the host
composited into a correct full-colour image of the live screen, and the reflashed package shows
v5.72 + the new build stamp in Unit Info.

## Recovery

A bad address in grab_routine can fault only the per-connection thread (one thread per
connection; the accept loop and the rest of the system are unaffected). The backstop is the
stock-package reflash, the recovery floor in [modification](modification.md).
